ARB Answer -- Why agent and not workflow?
  Multi_step queries (10% of 100K daily volume) require dynamic tool dispatch
  where the next tool cannot be determined until the prior result is seen
  (e.g. price lookup -> inventory check -> cross-product comparison in one turn).
  A deterministic workflow must hardcode all step paths at design time; when
  branching depends on intermediate content, the workflow either fails or
  requires unmanageable exception paths. The Agent archetype is the only option
  that handles all five query categories natively.

2. AGENT STRATEGY
=================================================================

Orchestration  : Keyword-based query classifier routes to four specialised tools
Tools          : price_lookup | order_status | policy_search | store_hours
Multi-step     : multi_tool dispatch (all four tools called; context merged)

Query Classification (five categories):
  price      -- keywords: price, cost, how much, cheaper, per ounce, cents
  order      -- keywords: order, tracking, deliver, ship, ORD-
  policy     -- keywords: return, refund, receipt, exchange, policy
  hours      -- keywords: open, close, hours, time, weekday, thanksgiving
  multi_step -- multiple categories matched OR 'compare' detected

Model Routing:
  gpt-4o-mini  -- 90% of queries (single-category: price, order, policy, hours)
  gpt-4-turbo  -- 10% of queries (multi_step category)
  Rationale   : gpt-4o-mini is 67x cheaper per call; quality delta < 0.05
                on golden dataset (within IN11 regression threshold)

Context Injection:
  Retrieval   : mock_retrieve() -- keyword overlap scoring, top-3 documents
  Token guard : Input truncated at 600 words (~800 tokens) to meet SLO
  Max output  : 150 tokens (concise retail answers)

ARB Answer -- What happens when the classifier misclassifies?
  The fallback category is 'multi_step', which calls all four tools and passes
  merged context to gpt-4-turbo. This incurs higher cost but never returns an
  empty answer. Misclassification rate is monitored via tool_call_accuracy metric
  (threshold >= 0.95). Weekly spot-checks flag systematic misrouting.

4. COST MODEL
=================================================================

Scale           : 100,000 queries/day | 30 days/month
Avg tokens      : ~200 input | ~60 output (gpt-4o-mini encoding)
Ceiling         : $1,500.00 / month

Monthly Spend Projections:
  Scenario 1 -- Baseline (all gpt-4-turbo, no optimisation)
    $11,400.00 / month   OVER CEILING  (7.6x over budget)

  Scenario 2 -- Model Routing (90% gpt-4o-mini / 10% gpt-4-turbo)
    $1,318.20 / month    WITHIN CEILING

  Scenario 3 -- Optimised (routing + 20% cache hit rate + 18% compression)
    $956.50 / month      WITHIN CEILING  <- Production target

  Worst-case (cache fails; routing + compression only)
    $1,195.62 / month    WITHIN CEILING

Optimisation Levers Applied:
  Model routing   -- gpt-4o-mini for 90% of queries  (saves ~$10,000/mo vs baseline)
  Response cache  -- 20% hit rate on repeat retail queries (saves ~$362/mo)
  Prompt compress -- 18% input token reduction via context trimming (saves ~$36/mo)

ARB Answer -- Worst-case monthly spend if cache fails?
  $1,195.62/month -- still within the $1,500 ceiling because model routing
  (the primary lever) remains active. Cache failure alone cannot breach budget.
  Budget alert fires at $50/day (day-over-day 120% threshold) to catch spikes early.

5. DEPLOYMENT MODEL
=================================================================

Full plan: capstone_deployment_model.txt

CI/CD Pipeline by Change Type:
  System prompt change   : PR -> eval gate -> staging -> canary 5%  -> prod
  Model version change   : PR -> regression (IN11) -> staging -> canary 10% -> prod
  RAG config change      : PR -> retrieval eval (hit_rate_at_3, MRR) -> staging -> prod
  Tool logic change      : PR -> unit tests + tool_call_accuracy -> staging -> prod

Pre-Deployment Gate (ALL must pass before canary):
  faithfulness >= 0.85  |  answer_relevancy >= 0.75  |  toxicity < 0.10
  hit_rate_at_3 >= 0.75  |  task_success_rate >= 0.90  |  tool_call_accuracy >= 0.95
  P95 latency < 3000ms (measured on staging under synthetic load)

Canary Progression:
  5% traffic -> 30 min monitor -> 25% -> 2 hrs monitor -> 100% -> 24 hr monitor

Rollback Triggers:
  Any metric below threshold in rolling 1-hour window
  P95 latency > 3000ms for 3 consecutive minutes
  Error rate > 2% for 5 consecutive minutes
  Daily spend > $60 (120% of expected $50/day budget)

Rollback Procedure (target: < 30 minutes):
  T+00  On-call declares rollback; updates incident channel
  T+05  Feature flag flipped -- 100% traffic routed to previous version
  T+10  Langfuse metrics confirmed returning to pre-deploy baseline
  T+20  Stakeholders notified; post-incident review ticket opened
  T+30  Rollback complete; production stable on previous version

ARB Answer -- Rollback for faithfulness drop 0.88 -> 0.76?
  0.76 is below the 0.85 threshold. The Langfuse rolling 1-hour alert fires
  automatically. On-call flips the feature flag at T+05. Total elapsed time
  to stable production: under 30 minutes. The prompt change is reverted in
  the configuration store and blocked from re-deploy until root cause is fixed.

Risk                Likelihood  Impact    Detection                         Owner
--------------------------------------------------------------------------------
hallucination       Medium      High      Faithfulness < 0.85; Langfuse     ML Platform
prompt_injection    Medium      High      Output monitoring; red-team/mo    Security
pii_leakage         Low         Critical  PII regex log scan; data masking  Data Governance
model_drift         Low         High      Weekly IN11 benchmark; 7-day trend ML Platform
cost_overrun        Medium      High      Daily spend alert at 70%/85%      Eng Manager
dependency_failure  Low         Critical  Health check; span error rate > 0  ML Platform

Key Controls:
  Hallucination   : RAG context grounding; faithfulness gate >= 0.85 blocks deploy;
                    human review triggered when score < 0.70
  Prompt Injection: System prompt hardening (IN08); input sanitisation; output validation
  PII Leakage     : Full query never logged; order IDs hashed; annual GDPR audit
  Model Drift     : Model version pinned; weekly regression gate; alert on 7-day drop > 0.05
  Cost Overrun    : Hard spend cap with fallback; auto-downgrade to gpt-4o-mini under load
  Dependency Fail : Circuit breaker (IN07); cached response fallback; secondary model endpoint

ARB Answer -- Detection and response plan for silent quality drift?
  Detection : Weekly automated benchmark (IN11) computes 7-day rolling faithfulness.
  Threshold : Alert fires when rolling avg drops > 0.05 from the post-deploy baseline.
  Response  : New deployments auto-blocked; on-call reviews Langfuse trace samples;
              if drift confirmed, rollback or model re-evaluation initiated within 24h.
